# capabilities

What the user *currently has* that lets them do things — licenses,
certifications, credentials, skills, hardware, access rights. The
inventory layer's first half (resources is the second).

The boundary that matters most: capabilities are what you have, not
what you wish you had. Aspirations are projects until they become real.
Don't write a capability row for a thing the user is studying for —
that's a project. Write the capability when they actually pass the
exam, land the cert, or buy the kit.

## Fields

- `id` — uuid, generated by the server.
- `name` *(required)* — short and specific. *"Commercial drone license
  (Part 107)"*, not *"drone stuff"*.
- `category` *(required)* — one of:
    - `license` — government-issued (driving, professional practice,
      firearm, etc.).
    - `certification` — formal third-party credential (AWS, language
      certs, first aid, etc.).
    - `credential` — academic degrees, professional memberships.
    - `skill` — competence not formally credentialed (Mandarin
      conversational, Python, sailing).
    - `hardware` — owned physical equipment relevant to projects
      (camera kit, workshop tools, vehicles).
    - `access` — memberships, residency rights, network access, paid
      subscriptions with strategic value.
    - `other` — escape valve; if used often the enum should grow.
- `description` — one or two sentences. Optional.
- `level` — one of `novice` | `intermediate` | `advanced` | `expert`.
  **Only allowed for categories `skill`, `credential`, `other`.**
  Licenses, certifications, hardware, and access are binary — you
  have them or you don't. The storage layer rejects level on those.
- `acquired_at` — when the user got it. Year-precision is fine.
- `expires_at` — for things with hard expirations (passports,
  professional licenses, certifications). Null when not applicable.
- `renewal_required` — boolean (default false). Hint to the scanner
  that this needs deliberate renewal action.
- `renewal_lead_time_days` — integer (default 90). How far in advance
  the scanner should start surfacing the expiration.
- `last_exercised_at` — most recent activity that touched this
  capability. Updated by the capture flow via
  `state_mark_capability_exercised`.
- `status` — one of:
    - `active` — currently held and usable (default).
    - `lapsed` — was held; has expired or atrophied past usefulness.
    - `retired` — user explicitly retired it (no longer claims it).
- `unlocks` — optional one-line note: what does this enable that
  matters? Used by `opportunity_research` to spot capability-leveraged
  opportunities.
- `notes` — free-form.
- `created_at`, `updated_at` — server-managed.

## Notes

- A capability that's never been exercised AND was created more than
  12 months ago is treated as atrophying by the scanner. Mark it as
  exercised when it gets real use, or set status=`lapsed` if it's
  genuinely gone.
- Hardware ≠ "stuff I own." Hardware is equipment that meaningfully
  enables a project. A toaster isn't a capability; a darkroom is.
- Don't track money in capabilities. Cash, runway, subscriptions with
  recurring cost — those go in `resources`. A *paid* gym membership
  is `access`; the *cost* of the membership is none of Cairn's
  business.
