# commitments

A commitment is something the user said they will do — for themselves
or for someone else. Different from a decision (which is *I chose X*)
and different from a project (which has scope and structure). A
commitment is a small, dated promise.

Most commitments are short-lived. Capture them so the system can notice
when they age out without resolution — that's signal about what the
user actually does versus what they say they'll do.

## Fields

- `id` — uuid, generated by the server.
- `what` *(required)* — one sentence describing the promise.
- `to_whom` *(required)* — who the commitment is to. `"self"` is the
  most common value; otherwise use a name or a stakeholder reference.
  Tracking this matters because broken promises to others corrode
  differently than broken promises to oneself.
- `due_at` — optional ISO timestamp. Null means open-ended.
- `status` — one of:
    - `open` — still owed (default).
    - `done` — fulfilled; `completed_at` and (optionally) `outcome` set.
    - `dropped` — explicitly abandoned; `dropped_reason` set.
- `related_project_id` — optional foreign key into `projects`. Use when
  the commitment is part of a larger project's work.
- `created_at`, `completed_at` — server-managed.
- `outcome` — one sentence at completion time. What actually happened?
  This is the field that ages best.
- `dropped_reason` — when status moves to `dropped`. A drop with no
  reason is data lost; the tool that drops a commitment asks for one.

## Transitions

- `state_complete_commitment(id, outcome?)` — open → done.
- `state_drop_commitment(id, reason?)` — open → dropped.
- `state_update(entity_type='commitment', id, patch)` — field edits
  (typo fixes, due-date pushes). Use sparingly; a commitment whose
  due-date keeps getting pushed is itself data.

## Notes

- The capture flow proposes commitments; the user confirms each one
  before it lands. Like everything else in the draft flow, the
  staging prevents the assistant from quietly accumulating obligations
  the user didn't actually take on.
- A commitment that's been `open` past its `due_at` for a long time is
  the scanner's strongest signal. It should ask whether to complete,
  drop, or push.
