# capability_requirements

Bridge table linking projects to the capabilities or resources they
need. The structural substrate for the scanner's "capability gap"
pattern.

A project can have many requirements. Each requirement points at
exactly one capability *or* one resource — not both.

## Fields

- `id` — uuid, generated by the server.
- `project_id` *(required)* — foreign key into `projects`.
- `capability_id` — foreign key into `capabilities`. Set when the
  project needs a specific capability.
- `resource_id` — foreign key into `resources`. Set when the project
  needs a minimum resource level.
- `required_level` — one of `novice` | `intermediate` | `advanced` |
  `expert`. Only valid when `capability_id` is set. Use only when
  level granularity actually matters; null is fine for binary
  capabilities.
- `required_amount` — number. Only valid when `resource_id` is set.
  Compared against `current_quantity` to decide satisfaction.
- `notes` — free-form.
- `created_at` — server-managed.
- `is_satisfied` — **computed at read time, not stored.** Reflects
  current inventory state:
    - For a capability requirement: the capability exists with
      `status='active'`, and (if `required_level` is set) its `level`
      meets or exceeds it (`novice < intermediate < advanced < expert`).
    - For a resource requirement: the resource exists and
      `current_quantity >= required_amount` (or `required_amount` is
      null).

## Validation enforced by the storage layer

- Exactly one of `capability_id` / `resource_id` must be set. Both or
  neither is rejected at write time.
- `required_level` only valid with `capability_id`.
- `required_amount` only valid with `resource_id`.
- The referenced project / capability / resource must exist.

## Notes

- `is_satisfied` is computed every read because inventory state
  changes constantly. Materializing it would just be a stale
  denormalization.
- The `scan_inventory` skill calls `list_capability_gaps` to surface
  unsatisfied requirements on active projects. Requirements on
  killed/dormant projects aren't actionable and are filtered out by
  default.
- Adding a requirement is a one-tool call (`state_add_capability_requirement`);
  removing one is `state_remove_capability_requirement`. There's no
  amend — the row is small enough that delete-and-recreate is the
  simpler convention.
