# interests

The topics this user's mind keeps returning to. Interests are
persistent attractors — they're not projects (no goal, no kill
criterion), not decisions (nothing was chosen), not Strategic Life
Units (those are universal categories; interests are particular to
this person).

The point of tracking them is to notice when a long-standing interest
goes quiet. That's signal too — sometimes it means the user has
moved on for good reasons, sometimes it means a thread worth picking
back up.

## Fields

- `id` — uuid, generated by the server.
- `topic` *(required)* — short label in the user's own framing.
- `description` — one or two sentences. What does this interest mean
  for this user? Why is it on the list?
- `status` — one of:
    - `active` — currently being pursued or thought about (default).
    - `dormant` — quiet for now; might come back.
    - `retired` — consciously moved on.
- `first_observed_at` — when this interest was first captured (often
  earlier than the row's `created_at` if seeded from older sources).
- `last_observed_at` — most recent time the assistant saw this
  interest referenced. Updated when new sources are attached.
- `source_count` — **derived**, not stored. Computed at query time as
  the count of `source_links` rows pointing at this interest. So the
  number is always consistent with the actual source attachments —
  there's nothing to keep in sync.
- `created_at`, `updated_at` — server-managed.

## Notes

- Tagging is reused. Interests link to Strategic Life Units via
  `entity_slu_links` (entity_type = `interest`), so the scanner can
  reason about which life areas an interest sits in.
- An interest with `status = active` and zero `source_count` over many
  months is the inverse of a project blindspot: it suggests the user
  *says* they're interested but the data doesn't show it. The scanner
  can ask whether to mark it dormant.
- Interests go through draft/commit like everything else. The
  assistant doesn't autonomously decide what the user is interested
  in.
