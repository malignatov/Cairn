# slu_assessments

One row per (life_unit, assessment session) — the user's importance,
satisfaction, and rough time spent on a single Strategic Life Unit at
one point in time. Together, the rows from one `session_id` form a
portfolio snapshot; the scanner walks across snapshots to detect drift.

## Fields

- `id` — uuid, generated by the server.
- `life_unit_id` — foreign key into `life_units`.
- `assessed_at` — server-set timestamp.
- `importance` *(required)* — integer 0–10. How much does this matter to
  the user, right now?
- `satisfaction` *(required)* — integer 0–10. How is it going, right now?
- `hours_per_week` — non-negative number, optional. A rough estimate. The
  skill prompts for one — better to ask than skip.
- `quadrant` — computed at write time from importance and satisfaction
  with threshold 6+:
    - `upper_right` — high importance, high satisfaction (going well)
    - `upper_left`  — high importance, low satisfaction (urgent attention)
    - `lower_right` — low importance, high satisfaction (maybe trim)
    - `lower_left`  — low importance, low satisfaction (kill candidates)
- `notes` — optional free text from the user.
- `session_id` — groups all SLU ratings from one assessment.

## Notes

- Trust this table for past state; don't rewrite old assessments. If the
  user wants to revise their view, run a fresh assessment.
- The quadrant boundary is fixed (6+) for v3. Personalized thresholds
  await enough data to inform them.
- A partial assessment (some SLUs rated, then paused) is still valuable.
  The scanner gracefully handles incomplete snapshots.
