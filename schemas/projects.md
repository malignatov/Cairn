# projects

A project is anything the user is actively pursuing, deliberately pausing,
or has explicitly killed. Use sparingly — track real commitments, not stray
ideas.

## Fields

- `id` — uuid, generated by the server.
- `name` *(required)* — short, recognizable label. Avoid sentence-length names.
- `description` — one or two sentences. What is this, in the user's own words?
- `status` — one of:
    - `active` — being worked on now.
    - `snoozed` — paused with intent to resume (set `snooze_until`).
    - `dormant` — backburner; no fixed wake date.
    - `killed` — abandoned; do not propose reviving without strong signal.
- `created_at` — set by the server.
- `last_touched` — auto-updated on any `state_update`.
- `snooze_until` — ISO timestamp; only meaningful when `status = snoozed`.
- `kill_criteria` — under what conditions should this be killed? Write this
  at the start, when judgment is unclouded by sunk cost.
- `revive_criteria` — what would have to be true for a `killed` or `dormant`
  project to come back? Often the most useful field.

## Notes

- Projects don't have owners, deadlines, or sub-tasks in v1. If the user
  wants to track tasks within a project, that's a different system.
- A project's death is information. Record `kill_criteria` early and revisit
  it later rather than litigating each judgment fresh.
