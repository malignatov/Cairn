# drafts

The capture flow's staging table — and the system's structural defense
against an LLM recording things the user didn't quite mean. When the
assistant extracts a project, decision, commitment, stakeholder, or
interest from a conversation, it does **not** write that straight to the
canonical table. It writes a *draft* here first, shows it to the user,
and only commits once the user says so.

A draft is something the **user** said, held until they confirm it.
Contrast with `suggestions`, which holds things the **assistant**
surfaced proactively (via `opportunity_research` / the scanners) and is
asking the user to consider. Different origin, different review path:
drafts are confirmed inline during capture; suggestions are triaged in
the inbox. The word "proposal" used to name this table through v6.1; it
was renamed to `drafts` to end the near-synonym collision with
`suggestions`.

Drafts are created with `state_draft`, edited with `state_amend_draft`,
and resolved with `state_commit_draft` or `state_reject_draft`. Nothing
here affects canonical state until a commit.

## Fields

- `id` — uuid, generated by the server.
- `entity_type` *(required)* — the type the draft will become once
  committed: one of `project`, `decision`, `commitment`, `stakeholder`,
  `interest`, `capability`, `resource`. (Not `suggestion` — those never
  go through the draft flow.)
- `data` *(required)* — a JSON object holding the full draft of the
  entity: every field the canonical row will have. Validated against the
  target type's rules at **commit** time, not draft time, so a
  half-formed draft can be amended into shape before it lands. May carry
  a `slu_links` array; on commit the `entity_slu_links` rows are created
  alongside the entity.
- `status` — one of:
    - `pending` (default) — staged, awaiting the user.
    - `amended` — edited via `state_amend_draft`; still awaiting commit.
    - `committed` — written to the canonical table. Terminal.
    - `rejected` — the user declined it. Terminal.
- `created_at` — set by the server.
- `committed_at` — set when the draft is committed; null otherwise.
- `rejection_reason` — the user's reason, captured on reject. Optional
  but worth recording.
- `session_id` — groups all drafts from one capture run (any string the
  skill generates, e.g. `capture-2026-05-25-...`). Lets the user review
  a whole capture as a unit.
- `committed_entity_id` — the id of the canonical row this draft became,
  set on commit. The link back from staging to the real entity.

## Notes

- Only `pending` and `amended` drafts can be committed, rejected, or
  amended. The two terminal states (`committed`, `rejected`) reject
  further transitions — a draft's history is immutable once resolved.
- A healthy DB has few or zero `pending` drafts sitting around: capture
  runs create them and resolve them in the same session. A pile of stale
  pending drafts means a capture flow was abandoned mid-way.
- Drafts are not full-text indexed. They're transient staging, not part
  of the searchable knowledge base — only what they become (the
  committed entity) is.
