# principles

The user's ranked operating rules — the explicit set of principles they
evaluate their decisions against, like a personal constitution checked at
decision time.

Principles are **prescriptive**: they govern how the user *chooses*. This is
a different layer from PERMA-V (`great_life_dimensions`), which is
**descriptive** — it measures how life is *going*. No overlap. A principle
may optionally relate to PERMA-V dimensions or echo Cairn's own
`constitution://main` (same values, two audiences: the constitution guides
the assistant's behavior, principles govern the user's decisions) — that
coherence is intentional, not duplication.

The model is **ranked and soft**. Rank 1 is highest. When two principles
conflict, the higher-ranked one should win — but the user can override a
principle, and an override requires justification. The system never blocks a
decision on principle grounds. Crucially: in a ranked system *some
violations are correct* — overriding a lower principle to honor a higher one
is the hierarchy working as designed, not a failure. (See
`decision_principle_evaluations` for the three-case model.)

No pre-seeding. The user brings their own principles via `define_principles`;
Cairn ships none. The system never infers principles from behavior, never
proposes a ranking, and never judges whether a principle is "good."

## Fields

- `id` — uuid, generated by the server.
- `name` *(required)* — short and memorable, e.g. "Rectitude" or
  "Depth over breadth".
- `description` *(required)* — one or two sentences in the user's own words.
  Captured faithfully; never paraphrased or "improved."
- `rank` *(required)* — integer, 1 = highest priority. **Unique among
  active principles.** Inserting at an occupied rank shifts the others down;
  re-ranking shifts the affected block. Uniqueness is enforced in
  application logic (not a DB constraint) because retired rows keep their
  historical rank value.
- `provenance` — optional free-text note on where the principle comes from
  ("Bushido — canonical", "personal addition"). Not enforced.
- `status` — `active` (default) | `retired`. Retiring frees the rank among
  the active set and leaves a gap the user may compact later. Rank
  uniqueness applies only to active principles.
- `created_at`, `last_revised_at` — timestamps.
- SLU tags via the shared `entity_slu_links` table (`principle` is a valid
  link entity_type). Most principles are cross-cutting; tag only where the
  connection is clear and narrow.

## Operations

- Create / read / edit go through the generic surface
  (`state_write("principle", …)`, `state_query`, `state_update`). A
  `state_update` to a principle requires a `rationale` in the patch and is
  logged as a `rewritten` revision; rank and status changes are rejected
  here.
- **Re-ranking** must go through `state_rerank_principle(id, new_rank,
  rationale)` — it performs the atomic reordering and logs a `reranked`
  revision. Never change `rank` via a plain update.
- **Retiring** goes through `state_retire_principle(id, rationale)`, logging
  a `retired` revision.

## Revision history (`principle_revisions`)

Every change to a principle is appended to `principle_revisions` — an
append-only log of `created` | `rewritten` | `reranked` | `retired` events,
each with a required `rationale` and the `prior_state` / `new_state` JSON
preserved. Amending a principle is ceremonial, like amending a constitution:
revisable, but with the history kept intact. A persistent divergence between
behavior and a principle isn't always the user failing the principle —
sometimes it's the user outgrowing it, and the log holds that story.

## Notes

- A healthy active set has contiguous ranks 1..N with no duplicates.
- Retired principles are excluded from the live hierarchy but preserved for
  history; they can still be referenced by past evaluations.
