# stakeholders

The people in the user's life that the system tracks deliberately —
not everyone they ever mention. Stakeholders are the named anchor
points for relationship-side blindspot detection: who hasn't been in
touch lately, who's owed a follow-through, who's central to a given
project.

This table is deliberately opt-in. The capture flow proposes
stakeholders one at a time and the user confirms each. The assistant
does not auto-extract people from chat into this table — that's the
boundary between a useful memory and surveillance.

## Fields

- `id` — uuid, generated by the server.
- `name` *(required)* — how the user refers to this person.
- `role` — optional one-phrase descriptor ("partner", "old colleague",
  "neighbor with the dog"). The user's words, not a category.
- `relationship_strength` — optional, one of `close`, `regular`,
  `weak`. Used as a weighting signal by the scanner; don't worry about
  precision.
- `last_contact_at` — ISO timestamp of the most recent contact event.
  Null when the stakeholder has been added but no contact has been
  recorded yet.
- `owed_action` — optional one-sentence note of what the user owes
  this person (a follow-up, a thank-you, a book they promised to
  lend). Null when nothing is outstanding.
- `notes` — free-form notes. `state_update_contact` appends timestamped
  entries here as the contact log builds up.
- `created_at`, `updated_at` — server-managed.

## Transitions and helpers

- `state_update_contact(stakeholder_id, when?, note?)` — record a
  contact event. Updates `last_contact_at`, appends a timestamped line
  to `notes` if given. Defaults `when` to now.
- `state_link_stakeholder_to_project(stakeholder_id, project_id, role_in_project?)`
  — many-to-many tag connecting a person to projects they're involved
  in.
- `state_list_overdue_contacts(threshold_days)` — stakeholders whose
  `last_contact_at` is older than the threshold (or null). The relationship
  side of blindspot scanning.

## Notes

- Deactivation isn't a thing here; if a relationship ends, set
  `relationship_strength` to `weak` or update the notes — don't
  delete. Lineage is part of the data.
- The contact log lives in `notes` as appended lines. If that gets
  unwieldy, a dedicated `contact_events` table is the natural next
  step — but not yet.
