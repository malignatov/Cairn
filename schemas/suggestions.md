# suggestions

The output of `opportunity_research` and any other proactive skill: a
concrete proposed action, grounded in the user's own data and (where
applicable) external evidence. Unlike the `drafts` table — which is
the capture flow's safety net for projects and decisions — this table
holds things the system thought of on its own and is asking the user to
consider.

Suggestions are written directly with `state_write("suggestion",
…)`. The validation here is structural, not stylistic: it stops the
table from filling up with disconnected generic noise.

## Fields

- `id` — uuid, generated by the server.
- `created_at` — set by the server.
- `kind` *(required)* — one of:
    - `opportunity` — a specific action with potential upside.
    - `risk` — a non-obvious downside materializing in the wider world.
    - `question` — something the user should think about, framed as a
      question rather than an action. Use this when confidence is low
      or the topic is sensitive.
    - `pattern` — a pattern noticed in the user's own data (e.g.
      "three decisions in the last month all defer this same call").
    - `blindspot` — a stated-vs-revealed gap or aging-without-progress
      observation, derived from the user's data alone.
- `content` *(required)* — the proposed action in concrete prose. Not
  "consider exploring X"; rather "talk to person Y about X within two
  weeks" or "log a decision about whether to add Q to project P".
- `rationale` *(required)* — which entity this connects to, what
  evidence supports it, why now.
- `linked_evidence` *(required)* — JSON object with two arrays:

    ```json
    {
      "internal_anchors": [
        { "entity_type": "project", "entity_id": "...", "why_anchored": "..." }
      ],
      "external_sources": [
        { "url": "...", "claim_supported": "...", "accessed_at": "..." }
      ]
    }
    ```

  Rules enforced by the storage layer:
    - `internal_anchors` must contain at least one entry. A suggestion
      with zero anchors is rejected at write time. This is the
      structural defense against generic noise.
    - `external_sources` must contain at least one entry when
      `kind` is `opportunity` or `risk`. May be empty for
      `pattern`, `blindspot`, and `question` (those can be derived
      entirely from internal data).
    - Each anchor needs `entity_type` and `entity_id`.
    - Each external source needs a `url`.

- `status` — one of `pending` (default) | `accepted` | `rejected` |
  `deferred`. Transition via `state_disposition_suggestion`.
- `disposition_reason` — captured when the user accepts, rejects, or
  defers. A rejection without a reason is signal lost; the inbox skill
  asks for one.
- `dispositioned_at` — set when the status leaves `pending`.
- `session_id` — groups suggestions from one `opportunity_research`
  invocation.

## Notes

- Suggestions are not editable in v1 of this layer; once written
  they're either dispositioned or they sit pending. To "amend" one,
  reject it and write a new one.
- Past rejected suggestions are signal. Future research runs read them
  to avoid re-pitching suggestions the user has already turned down.
